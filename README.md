## 개요
- **프로젝트 성격**: KH정보교육원 국비 팀 프로젝트(3인)
- **주요 역할**: 전산 시스템 개발 (시약 현황 관리 API 구축, 사용 등록 및 승인 워크플로우 구현)
- **작업 기간**: 2025.06 ~ 2025.07(5주)

## 핵심 구현 및 개발 포인트
- **대용량 데이터 매핑 및 가독성 최적화**: 복잡한 시약 데이터 및 연구 관리 테이블 레이아웃을 설계하고, MyBatis를 통해 조회된 데이터를 화면단에 유기적으로 매핑하여 데이터 조회 편의성을 높였습니다.
- **결재 프로세스 기반의 데이터 제어 흐름 구현**: 시약 사용 승인과 반려 절차를 처리할 때, 화면의 모달(Modal) 창과 데이터베이스(MySQL)의 쿼리 실행 순서를 연결했습니다. '반려'를 하면 사유 기록과 함께 결재 상태만 바꾸고, '사용 승인'을 하면 결재 상태가 바뀌면서 실제 시약 재고 테이블에서 승인된 수량만큼 재고가 바로 깎이도록 코드를 짰습니다. 이를 통해 화면과 데이터베이스의 데이터가 정확하게 일치하는 안정적인 관리 흐름을 구현했습니다.
- **MyBatis 동적 쿼리를 활용한 복합 검색**: 다중 조건 필터링 기능 구현 시, 서버단에서 파라미터 상태를 검증하고 MyBatis 동적 태그를 활용해 효율적으로 데이터를 필터링하여 시스템 운영 안정성을 확보했습니다.

## 트러블 슈팅
- **[현상]** 결재 시스템을 테스트하던 중, 관리자가 문서 결재나 시약 사용/추가 요청에 대해 '승인' 또는 '반려'를 눌렀을 때, 결재 상태는 바뀌지만 실제 시약 재고 데이터가 같이 수정되지 않거나 일부 내역이 저장되지 않는 오류 발생
- **[원인]** 승인/반려 처리를 하는 코드와 시약 테이블의 재고량을 조절하는 코드가 하나의 묶음(트랜잭션)으로 처리되지 않아, 처리 도중 문제가 생기면 한쪽 데이터만 바뀌는 현상이 생겼기 때문임.
- **[해결]** 결재 종류(문서, 시약 추가, 시약 사용)에 맞춰 데이터 처리 흐름을 다시 정리했습니다. '반려'를 할 때는 사유를 적는 칸과 함께 상태만 업데이트하도록 나누었고, '사용 승인'을 할 때는 결재 상태가 바뀌는 것과 동시에 시약 재고 테이블에서 승인된 수량만큼 재고가 바로 깎이도록 순서를 맞춰서 문제를 해결했습니다.
- **[배운점]** 한 번의 클릭으로 여러 테이블의 데이터가 한꺼번에 바뀌어야 하는 결재 시스템에서는, 데이터가 저장되고 수정되는 순서를 명확하게 제어해 주는 것이 프로그램의 안정성에 가장 중요하다는 것을 배웠습니다.
